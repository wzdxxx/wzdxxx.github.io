设计模式

发表于 2025-04-18 更新于 2025-04-21 分类于 代码质量 本文字数： 3.5k 阅读时长 ≈ 13 分钟

1 六大软件设计原则

1.1 开闭原则

对扩展开放，对修改关闭。可以用接口和抽象类实现。

1.2 里氏代换原则

子类可以扩展父类的功能，但不能改变父类原有的功能。否则可能会使继承体系的可复用性变差。

1.3 依赖倒转原则

高层模块不应该依赖低层模块，两者都应该依赖其抽象。

1.4 接口隔离原则

使用多个隔离的接口，比使用单个接口要好，降低类之间的耦合度。

1.5 迪米特法则

又叫最少知道原则，一个实体应当尽量少地与其他实体之间发生相互作用，使得系统功能模块相对独立。

1.6 合成复用原则

尽量使用合成/聚合的方式，而不是使用继承。

2 创建型模式

提供了一种在创建对象的同时隐藏创建逻辑的方式，而不是使用 new 运算符直接实例化对象。这使得程序在判断针对某个给定实例需要创建哪些对象时更加灵活。

2.1 单例模式

一个单一的类负责创建自己的对象，同时确保只有单个对象被创建。

  • 优点
    • 内存中只有一个实例，减少内存消耗。
    • 避免资源的多重占用。
  • 缺点
    • 没有接口，不能继承。
    • 与单一职责原则冲突。

2.1.1 线程不安全的饿汉式实现

  • 线程不安全
  • 不延迟加载
  • 被序列化和反射破坏

2.1.2 线程不安全的懒汉式实现

  • 线程不安全
  • 延迟加载
  • 被序列化和反射破坏

2.1.3 线程安全的懒汉式实现

  • 线程安全
  • 延迟加载
  • 被序列化和反射破坏

2.1.4 双重校验锁

  • 线程安全
  • 延迟加载
  • 被序列化和反射破坏

2.1.5 静态内部类

  • 线程安全
  • 延迟加载
  • 被序列化和反射破坏

2.1.6 枚举

  • 线程安全
  • 不延迟加载
  • 不被序列化和反射破坏

2.2 工厂方法模式

可以将对象的创建逻辑封装在一个工厂类中，而不是在客户端代码中直接实例化对象，这样可以提高代码的可维护性和可扩展性。

  • 优点
    • 调用者只需要知道对象的名称即可创建对象。
    • 扩展性高，如果需要增加新产品，只需扩展一个工厂类即可。
    • 屏蔽了产品的具体实现，调用者只关心产品的接口。
  • 缺点
    • 每次增加一个产品时，都需要增加一个具体类和对应的工厂，使系统中类的数量成倍增加，增加了系统的复杂度和具体类的依赖。

2.4 抽象工厂模式

围绕一个超级工厂创建其他工厂。负责创建一个相关对象的工厂，不需要显式指定它们的类。每个生成的工厂都能按照工厂模式提供对象。抽象工厂模式提供了一种创建一系列相关或相互依赖对象的接口，而无需指定具体实现类。

  • 优点
    • 确保同一产品族的对象一起工作。
    • 客户端不需要知道每个对象的具体类，简化了代码。
  • 缺点
    • 扩展产品族非常困难。增加一个新的产品族需要修改抽象工厂和所有具体工厂的代码。

2.5 原型模式

通过复制该原型对象来创建一个和原型对象相同的新对象,用于创建重复的对象，同时又能保证性能，就是创建对象的克隆。

  • 优点
    • 性能提高
    • 避免构造函数的依赖
  • 缺点
    • 配备克隆方法需要全面考虑类的功能，对已有类可能较难实现，特别是处理不支持串行化的间接对象或含有循环结构的引用时。
    • 必须实现Cloneable接口。

2.6 建造者模式

将一个复杂对象的构建过程与其表示相分离，从而可以创建具有不同表示形式的对象。

  • 优点
    • 分离构建过程和表示，使得构建过程更加灵活，可以构建不同的表示。
    • 可以更好地控制构建过程，隐藏具体构建细节。
    • 代码复用性高，可以在不同的构建过程中重复使用相同的建造者。
  • 缺点
    • 如果产品的属性较少，建造者模式可能会导致代码冗余。
    • 增加了系统的类和对象数量。

3 结构型模式

关注对象之间的组合和关系，旨在解决如何构建灵活且可复用的类和对象结构。

3.1 代理模式

通过引入一个代理对象来控制对原对象的访问。

  • 优点
    • 代理模式将访问控制与业务逻辑分离。
    • 可以灵活地添加额外的功能或控制。
    • 可以智能地处理访问请求。
  • 缺点
    • 增加了代理层可能会影响请求的处理速度。
    • 实现起来可能较为复杂。

3.2 适配器模式

通过一个中间件（适配器）将一个类的接口转换成客户期望的另一个接口，使原本不能一起工作的类能够协同工作。

  • 优点
    • 促进了类之间的协同工作，即使它们没有直接的关联。
    • 提高了类的复用性。
    • 增加了类的透明度。
    • 提供了良好的灵活性。
  • 缺点
    • 导致系统结构混乱，难以理解和维护。
    • 由于只能继承一个类，因此只能适配一个类，且目标类必须是抽象的。

3.3 装饰者模式

通过将对象包装在装饰器类中，以便动态地修改其行为。

  • 优点
    • 装饰类和被装饰类可以独立变化，互不影响。
    • 可以动态地添加或撤销功能。
    • 提供了一种继承之外的扩展对象功能的方式。
  • 缺点
    • 多层装饰可能导致系统复杂性增加。

3.4 桥接模式

用于把抽象化与实现化解耦，使得二者可以独立变化。

  • 优点
    • 提高了系统的灵活性和可维护性。
    • 可以独立地扩展抽象和实现。
    • 用户不需要了解实现细节。
  • 缺点
    • 增加了系统的理解与设计难度。
    • 在抽象层进行设计与编程。

3.5 外观模式

隐藏系统的复杂性，并向客户端提供了一个客户端可以访问系统的接口。

  • 优点
    • 客户端与子系统之间的依赖减少。
    • 子系统的内部变化不会影响客户端。
    • 隐藏了子系统的内部实现，只暴露必要的操作。
  • 缺点
    • 对子系统的修改可能需要对外观类进行相应的修改。

3.6 组合模式

用于把一组相似的对象当作一个单一的对象。

  • 优点
    • 客户端可以统一处理所有类型的节点。
    • 可以轻松添加新的叶子类型或树枝类型。
  • 缺点
    • 组件的声明是基于具体类而不是接口，这可能导致代码的灵活性降低。

3.7 享元模式

重用现有的同类对象，如果未找到匹配的对象，则创建新对象。

  • 优点
    • 通过共享对象，减少了内存中对象的数量。
    • 减少了对象创建的时间，提高了系统效率。
  • 缺点
    • 需要分离内部状态和外部状态，增加了设计和实现的复杂性。
    • 如果外部状态处理不当，可能会引起线程安全问题。

4 行为型模式

关注对象之间的通信和交互，旨在解决对象之间的责任分配和算法的封装。

4.1 模板方法模式

一个抽象类公开定义了执行它的方法的方式/模板。它的子类可以按需要重写方法实现，但调用将以抽象类中定义的方式进行。

  • 优点
    • 算法的不变部分被封装在父类中。
    • 子类可以扩展或修改算法的可变部分。
    • 减少代码重复，便于维护。
  • 缺点
    • 每个不同的实现都需要一个子类，可能导致系统庞大。

4.2 策略模式

定义了一系列算法或策略，并将每个算法封装在独立的类中，使得它们可以互相替换。

  • 优点
    • 可以在运行时根据需要切换算法。
    • 消除了复杂的条件语句。
    • 新增算法只需新增一个策略类，无需修改现有代码。
  • 缺点
    • 每增加一个算法，就需要增加一个策略类。
    • 策略类需要对外公开，以便可以被选择和使用。
